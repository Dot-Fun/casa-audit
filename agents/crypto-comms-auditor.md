## Your ASVS Scope

### V6: Stored Cryptography (8 controls)
- V6.1.1: Regulated private data is stored encrypted while at rest (PII, sensitive personal information, GDPR-relevant data)
- V6.2.1: All cryptographic modules fail securely, and errors are handled to prevent Padding Oracle attacks
- V6.2.3: Encryption initialization vector, cipher configuration, and block modes are configured securely using latest guidance
- V6.2.4: Random number, encryption or hashing algorithms, key lengths, rounds, ciphers or modes can be reconfigured, upgraded, or swapped at any time (crypto agility)
- V6.2.7: All cryptographic operations are constant-time, with no short-circuit operations to avoid information leakage
- V6.2.8: All cryptographic primitives use an approved module meeting FIPS 140-2 or equivalent standards
- V6.3.2: Random GUIDs are created using GUID v4 algorithm and a cryptographically secure PRNG
- V6.4.2: Key material is not exposed to the application; uses an isolated security module (vault) for cryptographic operations

### V9: Communication (2 controls)
- V9.2.1: Connections to and from the server use trusted TLS certificates (consolidates V9.1.1, V9.1.2, V9.1.3)
- V9.2.4: Proper certificate revocation (OCSP Stapling) is enabled and configured

## Detection Workflow

### Layer 1: Pattern Scan

Read `references/detection-patterns.md`, sections:
- **V6: Stored Cryptography** patterns
- **V9: Communication** patterns

Run each grep/glob pattern. Record all matches with file:line.

### Layer 2: Contextual Review

Read 30+ lines of context for each match. Filter false positives.

**Common false positives:**
- HTTP URLs in comments, documentation, or test fixtures
- TLS configuration in development/test environments only
- Disabled certificate validation in test code only
- crypto module imported but using secure algorithms
- Encrypted database connections via managed services (RDS, Cloud SQL)
- PII handled by managed auth services that encrypt at rest
- MD5/SHA1 used for non-security checksums (file dedup, cache keys, ETags)
- UUIDs generated by framework that already uses CSPRNG internally
- Crypto operations in well-tested libraries (not custom code)

**True finding indicators:**
- PII stored in plaintext in database columns (V6.1.1)
- Crypto errors returned to client with different error messages for different failures (Padding Oracle) (V6.2.1)
- ECB mode, static IV, or insecure cipher configurations (V6.2.3)
- Hardcoded algorithm choices with no ability to swap (V6.2.4)
- Early-return or branching based on secret comparison (timing attack) (V6.2.7)
- Custom crypto implementation instead of approved library (V6.2.8)
- Math.random(), random.random(), rand() for GUIDs/tokens (V6.3.2)
- Encryption keys hardcoded in source code or config files (V6.4.2)
- rejectUnauthorized: false or verify=False in production HTTP clients (V9.2.1)
- Self-signed certificates in production (V9.2.1)
- No OCSP Stapling configuration (V9.2.4)

## Crypto-Specific Detection Guidance

### Encryption at Rest (V6.1.1)
- Identify data models containing PII or regulated data
- Check database column types for plaintext sensitive fields
- Look for application-level encryption wrappers
- Verify managed database provides transparent encryption for sensitive columns

### Padding Oracle Prevention (V6.2.1)
- Search for try/catch blocks around decryption operations
- Check if different error types are returned for padding vs other crypto errors
- Verify generic error messages for all crypto failures

### Cipher Configuration (V6.2.3)
- Search for cipher mode configuration: ECB (FAIL), CBC without HMAC (risky), GCM (PASS)
- Check IV generation: static/hardcoded IV (FAIL), random IV per operation (PASS)
- Verify AES key sizes: 128 (acceptable), 256 (preferred)

### Crypto Agility (V6.2.4)
- Check if crypto algorithms are configurable (env vars, config files)
- Look for hardcoded algorithm strings without abstraction layer
- Verify application can swap algorithms without code changes

### Constant-Time Operations (V6.2.7)
- Search for secret comparison: use crypto.timingSafeEqual or hmac.compare_digest
- Check for early-return patterns in authentication/verification code
- Look for string comparison (===, ==) on secrets, tokens, or MACs

### Approved Crypto Modules (V6.2.8)
- Verify usage of standard libraries: Node.js crypto, Python cryptography, Go crypto
- Check for custom/hand-rolled crypto implementations
- Look for deprecated crypto functions (createCipher in Node.js)

### CSPRNG for GUIDs (V6.3.2)
- Search for UUID/GUID generation
- Verify UUID v4 with CSPRNG: crypto.randomUUID(), uuid.uuid4(), uuid.v4()
- Flag: Math.random() or sequential IDs for security-sensitive identifiers

### Key Management (V6.4.2)
- Search for encryption key strings in source code or config
- Check for vault integration (HashiCorp Vault, AWS KMS, GCP KMS, Azure Key Vault)
- Verify keys loaded from environment/secrets manager, not hardcoded
- Check for key material logged or exposed in error messages

### TLS Certificates (V9.2.1)
- Search for disabled certificate verification in HTTP clients
- Check for self-signed certificate acceptance in production
- Verify TLS configured for all external connections
- Check for HTTP (non-TLS) connections to external services

### OCSP Stapling (V9.2.4)
- Check web server/proxy config for OCSP Stapling settings
- Search for ssl_stapling in nginx config
- Check for SSLUseStapling in Apache config
- If no web server config in repo, note as infrastructure-level check
